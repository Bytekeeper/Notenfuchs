# Notenfuchs

Open-source grade-management tool for teachers (Klassen, Fächer, Noten). Focus is **grade entry and grade calculation** done well — that's the gap in existing tools. German context, self-hosted.

User-facing documentation (README.md and any other docs aimed at teachers, not developers) must be written in German — the audience is German-speaking teachers. Everything developer-facing stays English: CLAUDE.md itself, code (comments, Javadoc, identifiers), and commit messages.

## Tech stack

Quarkus 3.37.2 · Java 17+ · Hibernate ORM with Panache · RESTEasy Reactive (quarkus-rest) + Jackson · PostgreSQL · Flyway · Hibernate Validator · JUnit 5 · quarkus-oidc (alternative SSO auth) · quarkus-elytron-security-properties-file (default local auth) · Apache POI (poi-ooxml, xlsx export) · FastCSV (`de.siegmar:fastcsv`, roster CSV import/export) · quarkus-playwright (browser ITs). Postgres runs via Docker Compose. Group id `de.notenfuchs`.

The Maven Wrapper (`./mvnw`) is committed — **no local Maven install needed**, but Java 17+ is required.

## Commands

```bash
./mvnw test                    # unit tests (no DB needed — grade logic is pure)
./mvnw verify                  # also runs the Playwright browser ITs (needs Docker)
./mvnw quarkus:dev             # dev mode with hot reload (needs Postgres running)
./mvnw package                 # build
docker compose up -d postgres   # just the DB (for dev mode)
docker compose up --build       # full stack (app + Postgres)
```

App serves on `:8080`. Postgres connection is configured via env vars (`DB_USER`, `DB_PASSWORD`, `DB_URL`) in `application.properties`, defaulting to the docker-compose values.

## Architecture

### Package layout (`src/main/java/de/notenfuchs/`)
- `domain/` — Panache entities (the data model)
- `service/` — `GradeService` + its plain data carriers (`CategoryData`, `GradeData`, `SubjectAverageResult`); `CsvRosterService` + `RosterParseResult` for roster CSV (de)serialization; `PointsConversionService` + `PointsGradeBandData` for the points→grade Notenschlüssel conversion; `BehaviorGradeService` for the Verhaltensnoten cross-Fach student average + borderline check (see "Verhaltensnoten" below)
- `rest/` — one REST resource per entity + `ClassAveragesResource` for computed averages
- `dto/` — request/response records
- `web/` — server-rendered UI: one Qute/HTMX resource per entity (`ClassUiResource`, `SubjectUiResource`) + `GradeGridResource` for the grade-entry grid (also serves `/export` as an `.xlsx` download via Apache POI) + `BehaviorGridResource` for the Verhaltensnoten grid (see "Verhaltensnoten" below) + `ClassOverviewResource` for the ADMIN-only, read-only class-wide grade overview (see Authorization below). `ClassUiResource` also serves the roster CSV endpoints (`/classes/{id}/roster/export`, `/roster/import/preview`, `/roster/import`) — see the README's "Roster CSV format" section. Templates live in `src/main/resources/templates/`; the grid's keyboard/autosave JS is `static/js/notenfuchs.js`.
- `security/` — `CurrentUser`, a request-scoped facade over the `SecurityIdentity`/`UserInfo`/`JsonWebToken` for reading the logged-in teacher's subject/email/display name (works for both auth modes below); `OwnershipGuard`, the single point that enforces the multi-teacher access model (see Authorization below); `TeacherDirectoryRecorder`, which populates the `Teacher` directory the class/subject access UI picks teachers from (see "Class & subject access UI" below); `LocalAuthConfigSource` + `AuthConfigurationCheck`, which decide local-vs-OIDC and fail-fast-if-neither (see Authentication below)

### Data model
```
SchoolClass   → ClassTeacher (teacherSubject = a full co-owner's OIDC sub; a class can have several, see Authorization below)
SchoolClass   → Student (name is free-text; displayName = optional Kürzel/pseudonym)
SchoolClass   → Subject (has a GradeScale + roundingMode)
Subject       → SubjectTeacher (teacherSubject = a teacher of this Fach; a subject can have several, see Authorization below)
Subject       → GradeCategory (weightPercent, e.g. Schriftlich 50 / Mündlich 50)
GradeCategory → Assessment (a Leistung; factor, e.g. 2.0 = "counts double"; optionally pointsBased + its own roundingMode)
Assessment    → Grade (student_id, EITHER value NUMERIC(4,2) OR points NUMERIC(6,2), never both)
Assessment    → PointsGradeBand (only for a pointsBased Assessment; points → gradeValue, its Notenschlüssel)
GradeScale    (min, max, lowerIsBetter) — seeded "DE 1-6" via Flyway, shared across all teachers (not owned)
SchoolClass   → SchoolClass (optional predecessorClass, set only by the "copy into a new school year" action below)
SchoolClass   (optional halfYearCutoff DATE — display filter only, see "Halbjahr split view" below)
SchoolClass   (halfYearGradeDisplay WHOLE/HALF + optional halfYearTendencyThreshold — H1/H2 display only, see "Halbjahr grade display" below)
SchoolClass   → Student → BehaviorGrade ← Subject (one Verhaltensnote per student per Fach, see "Verhaltensnoten" below)
```

### Copy class into a new school year (`ClassUiResource#duplicate`)
The "Ins neue Schuljahr übernehmen" action on the class detail page (e.g. 8b → 9b) is an explicit **duplicate**, not an archive/lock: it creates a new `SchoolClass` (teacher supplies the new name + school year) owned by the current teacher, copying that class's Subjects (name, GradeScale, roundingMode), each Subject's GradeCategories (name, weightPercent), and Students (name, displayName) as a fresh, editable roster. Assessments and Grades are deliberately **not** copied — fresh start for the new year. The source class is completely untouched afterwards and stays a normal, editable class; there is no locking/archiving concept anywhere in this app (see ROADMAP.md's design principle). The source is resolved via `OwnershipGuard.requireClassAdmin` (duplication is an ADMIN-only action); the copy gets a fresh `ClassTeacher` row (role `ADMIN`) for whoever duplicated it and a fresh `SubjectTeacher` row per copied Subject — other teachers on the source class aren't carried over. `SchoolClass.predecessorClass` (nullable, `ON DELETE SET NULL`) links the copy back to its source — purely informational (future trend features), never consulted for access control, and set to null rather than blocking/cascading if the predecessor is later deleted.

### Notenschlüssel (points → grade per Leistung)
An `Assessment` can be marked `pointsBased` instead of taking a direct grade value — there's no upfront "max points" to configure. Its Notenschlüssel is a freely editable list of `PointsGradeBand` rows (`points` → `gradeValue`, an absolute points anchor, not a percentage), seeded with just two starting anchor bands (60 → best grade, 20 → worst grade on the subject's actual `GradeScale`, not hardcoded to 1-6) when an Assessment is first switched to points-based (`SubjectUiResource#seedDefaultBands`) — no Bundesland-specific scheme is imposed, and the teacher is expected to adjust the thresholds to their test and add/edit/delete bands freely afterward.

`Grade` stores **either** `value` **or** `points`, never both (enforced by a DB `CHECK` constraint) — which column is used is driven entirely by `assessment.pointsBased`. The grade for a points-based `Grade` is **never persisted**: `PointsConversionService.convert(points, bands, scale, roundingMode)` (a pure POJO service, like `GradeService`) derives it live from the raw points every time it's needed (grid render, cell-save response, row/column averages, xlsx export, `ClassAveragesResource`), by **linearly interpolating** between the two bands whose `points` bracket the entered points — the grade moves smoothly with points rather than jumping at each threshold. Points outside every band's range continue the linear trend of the two outermost bands on that side (so a 2-band Notenschlüssel is one straight line across the whole points axis, e.g. 60→1.0 / 40→3.5 extrapolates 33 points to ~4.4, not straight to the worst grade) and are clamped only at the underlying `GradeScale`'s absolute min/max — never before that, so bonus points can still climb all the way to the best grade and a very weak result can still fall all the way to the worst one. (A single-band Notenschlüssel can't define a slope, so it falls back to that one band's grade at/above its threshold and the scale's worst grade below it.) This is exactly why two default bands (60/20) are enough to start with — interpolation alone already produces a full smooth range; additional bands just let the teacher bend that line into a non-linear curve.

The interpolated result is rounded to **one decimal place** per `assessment.roundingMode` (`RoundingMode`, defaulting to `IN_FAVOR_OF_STUDENT`) — an independent per-Leistung choice mirroring `Subject.roundingMode`'s choice for the whole-grade average, editable via the same rename-form as the "Punktebasiert" toggle (`SubjectUiResource#renameAssessment`), only meaningful while `pointsBased` is true. `IN_FAVOR_OF_STUDENT` always rounds toward the better grade per `lowerIsBetter` (`FLOOR` on a lower-is-better scale, `CEILING` otherwise) — every fractional digit gets truncated in the student's favor, not just an exact half, unlike `GradeService`'s whole-number rounding of the same name — so a borderline points value never gets nudged to a worse grade than it earned. `COMMERCIAL` instead rounds with standard numeric half-up rounding, ignoring which direction that favors. This means editing the stored points, the Notenschlüssel bands, *or* the rounding mode immediately changes every average that depends on it — consistent with this app's anti-freeze design principle (see ROADMAP.md). `GradeService` itself stays completely unaware of points/bands/rounding mode: callers derive the grade value first, then feed it in as a normal `GradeData`.

In the grade-entry grid (`GradeGridResource`), a points-based column accepts raw points and shows the derived grade alongside it (e.g. "65 → 1"); switching an Assessment's points-mode (`SubjectUiResource#renameAssessment`) wipes its existing `Grade` rows, since a value entered in one mode has no meaning in the other. Points/bands/rounding-mode changes need coverage in `PointsConversionServiceTest` (band boundaries at exact thresholds, scale-agnosticism, both rounding modes) plus a Playwright IT in `GradeGridE2EIT` if the grid/web flow changes.

### Halbjahr split view (`GradeGridResource`)
`SchoolClass.halfYearCutoff` (nullable `DATE`, editable on the class detail page via `fragments/halfYearCutoff.html`) is **purely a display/query filter** on the grade grid — not a new grading concept. There is no Halbjahr entity, nothing is snapshotted, and every average still recomputes live from the same `Assessment`/`Grade` rows (see ROADMAP.md's anti-freeze design principle); changing or clearing the cutoff just changes how the *same* data is grouped on screen. When `halfYearCutoff` is null, `GradeGridResource#grid`/`#exportXlsx` render exactly as before this feature existed, via the original `loadGridData`/`buildWorkbook` code paths — completely untouched, for backward compatibility.

When a cutoff is set, `HalfYearAssessmentPartitioner` (a pure POJO service, unit-tested without a DB like `GradeService`/`PointsConversionService`) classifies each `Assessment` by `date` into `UNDATED`, `FIRST` (`date <= cutoff`, the boundary date itself) or `SECOND` (`date > cutoff`). The grid then renders, within the existing category grouping, up to four column regions left to right: **"Ohne Datum"** (undated Assessments — only rendered at all if at least one exists; visually flagged with a warning tint + a "kein Datum – zählt nur in die Jahresnote" hint, since these count only into the whole-year figure, never a half), **"1. Halbjahr"** and **"2. Halbjahr"** (each with its own subject-average column), and a final **"Jahr"** average column computed over every Assessment regardless of date — identical to the pre-Halbjahr single-average behavior. `data-col` indices for editable cells stay globally continuous across the Ohne Datum → H1 → H2 blocks (Jahr contributes no cells of its own, only an average column) so the existing keyboard nav/autosave in `notenfuchs.js` needed zero changes; the JSON cell-save response's average fields now include `h1RawAverage`/`h1FinalGrade`/`h2RawAverage`/`h2FinalGrade` alongside the always-present whole-year `rawAverage`/`finalGrade`, and each average `<span>` in the template carries a `data-scope` (`h1`/`h2`/`jahr`) so a single cell edit refreshes all of a student's average columns live via `updateAverageScope` in `notenfuchs.js`.

Crucially, `GradeService` itself is **never modified or made aware of Halbjahr**: `GradeGridResource` builds a `CategoryData` list restricted to one half's Assessments (a category with none of that half's Assessments graded is simply absent from the list, so `GradeService`'s existing empty-category normalization excludes it exactly as it already does) and calls `calculateSubjectAverage` on it like any other subject average — an H1/H2 average *is* an ordinary `GradeService` call, just over a pre-filtered subset. The xlsx export (`GradeGridResource#buildHalfYearWorkbook`) mirrors the same Ohne Datum/H1/H2/Jahr column structure as the HTML grid. Halbjahr changes need coverage in `HalfYearAssessmentPartitionerTest` (boundary date, undated, and — combined with `GradeService` — that a half's average equals calling `GradeService` directly on that date-filtered subset) plus an HTTP-level IT (`GradeGridHalfYearIT`) and a Playwright IT (`GradeGridHalfYearE2EIT`) if the grid/web flow changes.

### Halbjahr grade display: whole/half grades + tendency (`HalfYearGradeDisplayService`)
`SchoolClass.halfYearGradeDisplay` (`WHOLE`/`HALF`, default `WHOLE`) plus `SchoolClass.halfYearTendencyThreshold` (nullable `BigDecimal`, precision 3 scale 2, meaningful under *either* mode) control how an H1/H2 average cell is *displayed* — like `halfYearCutoff` above, this never changes the underlying calculation: `GradeGridResource` still computes the ordinary `rawAverage`/`finalGrade` via `GradeService`, and `HalfYearGradeDisplayService` (a pure POJO service, unit-tested without a DB) only decides how that pair becomes a display string. "Jahr" always stays a plain whole grade regardless of this setting, matching every Land's rules for the final report. The threshold is a raw deviation compared directly against a raw average (e.g. `0.1`) rather than a percentage a teacher would first have to convert to a fraction — simpler to reason about, and it's exactly the unit `tendencySuffix`/`label` already operate in internally.

Without a threshold configured, the two modes are simple and independent: `WHOLE` shows the bare `finalGrade`; `HALF` shows the raw average rounded to the nearest half-grade (`HalfYearGradeDisplayService#roundToHalf`). With a threshold configured, `HALF` doesn't reinvent its own scheme — it **reuses `WHOLE`'s exact tendency computation** (same anchor `finalGrade`, same threshold, same suffix direction via `tendencySuffix`) and adds one refinement: whenever that computation would produce a suffix (i.e. the raw average is outside `finalGrade`'s plain zone), `HALF` checks whether the raw average is *also* within that same threshold of the neighboring half-grade the suffix points toward — the half-grade adjacent to `finalGrade` in the numeric direction of the deviation. If so, that half-grade is shown bare instead of "finalGrade+suffix", since it's the more honest label. E.g. with a 0.1 threshold: 2.6 is outside whole grade 3's plain zone (leaning "+"/better) but well within 0.1 of the half-grade 2.5 — so `HALF` shows "2.5", while `WHOLE` (which has no such refinement) still shows "3+". A value in the genuine "murky middle" between a whole grade and the neighboring half-grade (e.g. 2.2 with a 0.1 threshold: outside grade 2's plain zone, but not yet close enough to 2.5 either) falls back to the same "finalGrade+suffix" both modes would otherwise show, e.g. "2-". The one limit this app enforces (settings-form HTML bounds, `min="0" max="0.49"`) applies identically to both modes: 0.5 is where adjacent whole grades' plain zones would start overlapping, not a HALF-specific cutoff - even a threshold approaching 0.49 stays well-defined in `HALF`, it just means the "murky middle" suffix case rarely triggers (the wide plain zone around `finalGrade` swallows almost everything, and the rest is close enough to the neighboring half-grade to refine into it).

This collapses what German regulations (e.g. Baden-Württemberg's NVO, Berlin's AV-Zeugnisse) describe as three separate interim-report conventions — plain "ganze Note", "ganze Note mit Tendenz" (+/-), and "halbe Note" — into one continuous scale: half-grade territory is simply the finer resolution reached once a whole-grade tendency would otherwise apply, rather than a competing, mutually exclusive display mode. Since the refinement is anchored on `finalGrade`, it's invariant to which tie-break the subject's own `RoundingMode` (`COMMERCIAL` vs `IN_FAVOR_OF_STUDENT`) picks at an exact `X.5` raw average — both converge on the same correct half-grade.

Changes here need coverage in `HalfYearGradeDisplayServiceTest` (the `roundToHalf` boundaries, both modes with no threshold, `WHOLE`'s tendency direction on both scale directions, and specifically `HALF`'s refine-to-half-grade behavior — both the "outside the plain zone but close to a half-grade" case and the "murky middle" fallback to a suffix) plus the HTTP-level `GradeGridHalfYearIT` and a Playwright IT (`GradeGridHalfYearE2EIT`) if the grid/web flow changes.

Changes here need coverage in `HalfYearGradeDisplayServiceTest` (the `roundToHalf` boundaries, the WHOLE/HALF mode split, the tendency-suffix direction on both scale directions, and specifically the half-grade-territory suppression above — both at an exact tie like 3.5 and at a non-tie like 2.6) plus the HTTP-level `GradeGridHalfYearIT` and a Playwright IT (`GradeGridHalfYearE2EIT`) if the grid/web flow changes.

### Verhaltensnoten (behavior grades)
A `BehaviorGrade` is one behavior/conduct grade per (student, Fach), entered directly by the teacher rather than derived from any `Assessment` — deliberately independent of `Grade`/`GradeService`: it never contributes to a subject's academic average, it only appears as its own separate figure on the Halbjahres-/Endjahreszeugnis. There is **no H1/H2 split** — unlike the Halbjahr grade-grid view above, a Verhaltensnote is a single always-current value the teacher updates before each report is printed, which is exactly this app's anti-freeze design principle applied to a new figure rather than a new exception to it (see ROADMAP.md). It reuses the owning `Subject`'s `GradeScale` rather than inventing a separate conduct scale, consistent with the scale-agnostic design principle below; a unique constraint on `(student_id, subject_id)` means editing a cell always overwrites in place.

`BehaviorGridResource` (`GET /classes/{id}/behavior-grid`) always renders a class-wide grid (students × every Fach of the class), unlike `GradeGridResource` which is scoped to one Fach at a time — any teacher with class access (owner, or a Fachlehrer teaching just one Fach in the class, see Authorization below) sees every Fach's column, a deliberate broadening of visibility beyond "subjects you teach" for this one figure. Cell edits POST to `/classes/{id}/behavior-grid/cell` (`studentId`, `subjectId`, `value`), validated against that Fach's own `GradeScale` bounds exactly like the grade grid's cell save — editing narrows back down to `requireTeachesSubject` for the specific Fach being written, so viewing stays class-wide but editing doesn't. The template reflects that split rather than just relying on the 404: each `CellView` carries a `taught` flag (`OwnershipGuard#teachesSubject`), and a Fach the viewer doesn't teach renders its column's `<input>`s `readonly` (`.grade-input-readonly` in `notenfuchs.css`) instead of a control that would silently fail to save — no `notenfuchs.js` changes needed, since a readonly input still receives focus (keyboard nav/`data-col` continuity untouched) but never changes value, so `saveCell`'s change-detection never fires a request for it.

The grid shows two different kinds of average, and they are **not computed the same way**:
- **Ø je Fach** (footer, one per column): a plain average across every student who has a Verhaltensnote in that one Fach, within that Fach's own `GradeScale` — this reuses `GradeService#calculateAssessmentAverage` unchanged (exactly the per-Leistung footer average pattern from the grade grid) and so gets a proper rounded final grade.
- **Ø** (rightmost column, one per row): a student's Verhaltensnoten across *all* their Fächer, which may each use a *different* `GradeScale` — there is no single scale to round a discrete final grade against. `BehaviorGradeService` (a pure, DB-free POJO service like `GradeService`) therefore only produces a raw numeric average, plus an `isBorderline` flag (true when the average's fractional part is within 0.4–0.6 of a whole number, e.g. 2.4–2.6 around 2.5) that highlights a student's row when a small change could tip which whole grade the value is closest to. The grid highlights only this row average (yellow tint, `.average-cell.borderline` in `notenfuchs.css`) — the per-Fach column average never gets this treatment, since it already has a real scale-bound final grade and isn't subject to the cross-scale ambiguity.

The grid's keyboard nav/autosave is a separate, smaller `initBehaviorGrid` in `notenfuchs.js` (scanning `.behavior-grid-root`, not `.grade-grid-root`) rather than a generalization of the grade grid's `initGradeGrid` — the wire field is genuinely `subjectId`, not `assessmentId`, and there's no points-based/Halbjahr-scope handling to carry over. `rest/BehaviorGradeResource` (`/api/behavior-grades`) mirrors `GradeResource` as the plain CRUD REST API for the same entity, per this codebase's "one REST resource per entity" convention (see Package layout above) — `OwnershipGuard#requireClassAccessBehaviorGrade`/`#requireTeachesBehaviorGrade` back both it and the grid's view/cell-save endpoints respectively. Verhaltensnoten changes need coverage in `BehaviorGradeServiceTest` (average rounding, the borderline boundary at every whole-number grade, not just around 2.5, null/empty handling), `OwnershipGuardIT` (class-wide view vs. teaches-subject-scoped edit, plus cross-tenant isolation), an HTTP-level IT (`BehaviorGridIT`: cell save/clear, both average kinds, cross-tenant 404s, cascade delete, and `gridPage_marksCellReadonlyForSubjectViewerDoesNotTeach` for the `taught`/readonly rendering above) and a Playwright IT (`BehaviorGridE2EIT`) if the grid/web flow changes.

### Key design decision — scale-agnostic grades
Grade values are stored as `NUMERIC` (BigDecimal), **never** a 1–6 enum. The scale lives in `GradeScale`. This is deliberate: a future 0–15 Punkte scale can be added **without a schema change or any change to the calculation math**. When touching grade logic, keep it scale-agnostic — read `lowerIsBetter` from the scale, never assume 1–6 or "lower is better".

### Grade calculation (`GradeService`)
Per student, per subject:
1. **Category average** = mean of the student's grades in that category, each weighted by `assessment.factor`.
2. **Subject average** = category averages combined by `weightPercent`, normalized over the sum of weights of **only the categories that actually have grades** (an empty category must not distort the result).
3. Expose the **raw average** (2 decimals) *and* a **final grade** rounded per `roundingMode`:
   - `COMMERCIAL` — round half up (2.50 → 3)
   - `IN_FAVOR_OF_STUDENT` — round half toward the better grade (consults `lowerIsBetter`; for DE 1-6: 2.50 → 2)

Use `BigDecimal` throughout — no `double`. `GradeService` is a pure POJO service over in-memory data so it's unit-testable without a DB; keep it that way.

### Authentication (local built-in, default) vs OIDC (alternative)
Notenfuchs needs exactly one of two things configured, never both at once: a local password (`NOTENFUCHS_PASSWORD`, the default turbo path — `docker compose up` and nothing else) or an OIDC issuer (`OIDC_ISSUER_URL`, for schools with existing SSO). `security/LocalAuthConfigSource` (a `ConfigSource`) is the single place this decision is made, derived purely from whether `NOTENFUCHS_PASSWORD` is set — not a separate `AUTH_MODE` flag.

Local auth uses `quarkus-elytron-security-properties-file`'s embedded realm (one fixed username, `lehrer` — a Java constant in `LocalAuthConfigSource`, not env-driven, since embedded-realm map keys aren't interpolated per-entry) plus Quarkus' built-in FORM authentication (`quarkus.http.auth.form.*`) — no user table, no registration, no password hashing. `quarkus.security.users.embedded.enabled` and `quarkus.http.auth.form.enabled` are `BUILD_AND_RUN_TIME_FIXED` (they gate CDI bean registration at augmentation time), so they're always `true` in `application.properties` — one built image must support either mode without a rebuild. What `LocalAuthConfigSource` actually drives at genuine `RUN_TIME`: the embedded user's password (the real value, or an unguessable per-process random UUID when `NOTENFUCHS_PASSWORD` is unset — never blank, which would otherwise let an empty password field log in), `quarkus.oidc.tenant-enabled` (forced `false` when local is active), and `quarkus.http.auth.permission.authenticated.auth-mechanism` (pinned to `form` or `code` — Quarkus's OIDC code-flow mechanism's literal scheme name, **not** `"oidc"`, see `OidcConstants.CODE_FLOW_CODE`). That pin is load-bearing, not cosmetic: both mechanisms are always structurally present in the built app, so without it they'd race to answer an unauthenticated request's challenge, and a wrong pin silently breaks login for whichever mode isn't pinned. `web/LoginUiResource` (`GET /login`, public) serves the login page standalone (not extending `base.html`, since there's no logged-in user yet); the form posts straight to Quarkus' own `/j_security_check` (`quarkus.http.auth.form.post-location`). `web/LocalLogoutResource` (`POST /local-logout`) calls the static `FormAuthenticationMechanism.logout(identity)` to clear the session cookie — a distinct path from `/logout`, which stays OIDC's own RP-initiated logout route (always registered regardless of `tenant-enabled`, so the two can't share a path). `base.html`'s nav picks the right logout link/form via `CurrentUser.localAuthActive()`.

`security/AuthConfigurationCheck` observes `StartupEvent` and, in `%prod` only (`LaunchMode.NORMAL`), throws if neither `NOTENFUCHS_PASSWORD` nor `OIDC_ISSUER_URL` is set — refuses to boot an unauthenticated instance rather than silently serving every teacher's data. `CurrentUser` itself needed **no changes** for local auth: `SecurityIdentity`/`isAuthenticated()`/`subject()` are populated identically by the elytron identity provider (principal name = the fixed username), and `UserInfo`/`IdToken` already fail closed to `null` when there's no OIDC session (see `CurrentUser`'s existing try/catch), so `displayName()` falls back to the username automatically.

Local-auth changes need coverage in `LocalAuthConfigSourceTest`/`AuthConfigurationCheckTest` (pure predicate unit tests — no `System.getenv` mocking; both classes expose a testable static method taking the raw env value as a parameter) plus `LoginIT` (HTTP-level: unauthenticated redirect, correct/wrong password, session cookie) and a Playwright `LoginE2EIT` if the login page/flow changes. Both ITs use a shared `web/LocalAuthTestProfile` (`@TestProfile`) to flip the `%test` blanket-permit bypass back to real enforcement for just those two classes — `@TestProfile` config overrides win over everything else and force a fresh augmentation, which is what actually lets `BUILD_AND_RUN_TIME_FIXED` properties differ per test class even though they can't differ per env var in a single built prod image.

### Authorization
Every path is authenticated by default via a global HTTP permission policy in `application.properties` (`quarkus.http.auth.permission.authenticated.paths=/*`, `policy=authenticated`) — not per-endpoint `@RolesAllowed` annotations. Only `/auth-callback,/logout,/login,/j_security_check,/q/health/*,/favicon.ico,/static/*` are public. This means a new REST or web resource is protected automatically with no annotation required; don't add `@PermitAll` without a specific reason. Either local auth or OIDC (see Authentication above) backs the login; `CurrentUser` (`security/`) exposes the logged-in teacher's identity regardless of which mode is active. The `%dev`/`%test` profiles disable the OIDC tenant and relax the policy to `permit` so local dev and `./mvnw test` don't need a live IdP or a configured password — see the README's Authentication section for details.

Authentication only proves *who* is logged in; **the multi-teacher access model** is a separate layer on top, with three tiers across two entities, both keyed on a teacher's OIDC `sub`. `ClassTeacher` attaches a teacher to a `SchoolClass` at one of two `ClassTeacherRole`s: `ADMIN` (today's full co-owner — roster read/write, class-wide settings, managing admins and the FACHLEHRER tier itself, deleting any Subject, and a read-only class-wide grade overview) or `FACHLEHRER` (class-level: can add a new Subject and delete/manage Subjects it personally teaches, but not administer the class). `SubjectTeacher` marks who teaches a specific `Subject` (subject-level Fachlehrer-tier; gates all Leistung-level access — categories, assessments, grades, subject config, renaming it — for that one subject, true regardless of `ClassTeacher` role, so an admin who doesn't personally teach a subject can't see its grades either) — and has **no admin override**: adding/removing a `SubjectTeacher` stays exclusively self-service by whoever currently teaches that Fach, a deliberate choice confirmed explicitly (an admin who doesn't teach a Fach can't add a colleague to it). Plain class-wide access (roster read, subject list, Verhaltensnoten) is deliberately **derived, not stored**: a teacher has it if they hold a `ClassTeacher` row of either role, or teach at least one of the class's subjects (`OwnershipGuard#hasClassAccess`) — this is what lets a Fachlehrer share their one Fach with a colleague and have that colleague land with exactly the class access they need, with no separate approval step or second row to keep in sync. Granting the class-level FACHLEHRER tier itself is different: it *is* a deliberate, ADMIN-only grant (`addClassTeacher`'s `role` param), precisely because it crosses into class-structural power (creating/deleting Fächer) that a single-subject share never did. `GradeScale` is shared reference data and is *not* owned. `security/OwnershipGuard` is the single place this is enforced: every REST/web endpoint that reads or writes an entity by id must resolve it via one of `OwnershipGuard`'s `require*` methods (never a raw `findById`) — `requireClassAccess` (view, either role or teaches a subject) / `requireClassTeacher` (either role — can add a Subject) / `requireClassAdmin` (ADMIN-tier only) for class-level, `requireClassAccessSubject` (metadata) / `requireTeachesSubject` (Leistung-level + renaming, self-service) / `requireCanDeleteSubject` (admin deletes any Subject, a class-level Fachlehrer only one it teaches, a subject-only Fachlehrer none) for subject-level, `requireTeaches*` for every other Leistung-level entity (category/assessment/grade/points-grade-band), `requireClassAccessStudent`/`requireRosterManageStudent` for students (roster mutation stays ADMIN-only, unaffected by the FACHLEHRER tier), `requireClassAccessBehaviorGrade`/`requireTeachesBehaviorGrade` for Verhaltensnoten (viewable class-wide, editable only for a subject you teach), and `requireClassAdminTeacher`/`requireTeachesSubjectTeacher` for managing `ClassTeacher`/`SubjectTeacher` rows themselves (see "Class & subject access UI" below) — and every list endpoint must go through `listAccessibleClasses`/an equivalent scoped query (never `listAll()`). A foreign or unknown id both come back as 404 (`NotFoundException`), deliberately indistinguishable, so a teacher can't probe for the existence of another teacher's data. `CurrentUser.effectiveSubject()` — not `subject()` — is what call sites pass in: it falls back to the fixed `CurrentUser.DEV_USER_SUBJECT` ("dev-user") constant when there's no authenticated session, so access still works in `%dev`/`%test` where authentication is bypassed entirely (see Authentication above). New entities scoped by class or subject must add a `require*` method to `OwnershipGuard` rather than checking access inline at the call site.

The backend gate is only half the story — server-rendered pages must also not *render* controls a viewer can't actually use, since a 404/400 from clicking a dead button is a worse experience than not showing it at all. `ClassUiResource#detail` computes `isClassAdmin` (`OwnershipGuard#isAdmin`), `isClassTeacher` (`OwnershipGuard#isClassTeacher`, either role), and `taughtSubjectIds` (which of this class's Subjects the viewer is a `SubjectTeacher` of), passing all three to `ClassPage/detail.html`: ADMIN-only sections (Schüler hinzufügen, roster CSV import, Lehrkräfte/`ClassTeacher` management, Klasseneinstellungen) are wrapped in `{#if isClassAdmin}`, "Neues Fach anlegen" in `{#if isClassTeacher}` (either role, matching `requireClassTeacher`), and `fragments/subjectList.html` shows a Subject's Noten/Ändern controls only when `taughtSubjectIds.contains(s.id)` and its Löschen control only when `isClassAdmin` (any Subject) or (`isClassTeacher` and taught) — mirroring `requireCanDeleteSubject` exactly. A class-access-only viewer still sees an untaught Subject listed (matches `requireClassAccessSubject`'s "metadata only" scope) but as plain text, not a dead link — unless they're an admin, in which case they still get a Löschen control on it despite not teaching it. Roster CSV *export* and the Verhaltensnoten link stay visible to every class-access holder, since both really are class-wide. Every fragment-returning mutation endpoint (`addSubject`/`deleteSubject`/`renameSubject`, `addStudent`/`deleteStudent`/`renameStudent`) must keep threading the same `isClassAdmin`/`isClassTeacher`/`taughtSubjectIds` data through its response — easy to miss since nothing forces it (these templates aren't `@CheckedTemplate`-typed), and an omission would wrongly hide controls from the very actor whose action just proved they're allowed to see them. The same pattern applies one level up: the class *list* (`/classes`, `fragments/classList.html`) shows every class the viewer has any access to via `OwnershipGuard#listAccessibleClasses` (including classes where they're only `FACHLEHRER`-tier or a subject-only Fachlehrer), so `ClassUiResource#list`/`create`/`delete`/`rename` all compute and pass `adminClassIds` (ids of classes the viewer administers, via `ClassTeacher.list(... role = ADMIN)`) so the template can hide Ändern/Löschen for classes they don't administer, the same "don't render a dead button" principle.

`fragments/subjectList.html` also shows which teacher(s) teach each Subject (`ClassUiResource#attachTeacherLabels`, batch-resolving `SubjectTeacher` rows against the `Teacher` directory onto `Subject#primaryTeacherLabel`/`#otherTeacherLabels`) - the first teacher inline, the rest behind a `<details>` disclosure (`+N weitere`), shown regardless of whether the viewer teaches that Subject themselves. This is deliberately *not* gated by `taughtSubjectIds` like the Noten/Ändern/Löschen controls above: it's what lets a class admin see who to contact per Fach, and cross-reference against the "Lehrkräfte" section (which only lists `ClassTeacher` rows) to notice which teachers have class access purely by teaching a Subject.

Since that cross-reference is passive (an admin only notices by comparing two sections themselves), `ClassUiResource#transitiveTeachers` does it for them: an ADMIN-only, read-only "Fachlehrer mit Klassenzugriff" section on the class page, directly below "Lehrkräfte", listing every teacher who has class-wide access (roster, Verhaltensnoten) *solely* by teaching one of the class's Subjects - i.e. every distinct `SubjectTeacher.teacherSubject` across the class's Subjects that has no `ClassTeacher` row of its own here - alongside which Subject(s) grant it. This exists because `SubjectTeacher` is deliberately self-service with no admin override (see above): a Fachlehrer can grant a colleague full class-wide visibility without the class admin ever approving it or necessarily knowing, and unlike that colleague formally joining a Fach they could already leak informally (print/share), this is genuinely new exposure the admin didn't grant. Purely informational, like the per-Fach display above - access is still granted/revoked on the Subject itself (`fragments/subjectTeachers.html`), not here. A teacher who both teaches a Subject *and* holds a `ClassTeacher` row is excluded (they're not "transitive" - already visible in "Lehrkräfte").

Both `ClassTeacher` and `SubjectTeacher` have management UI (see "Class & subject access UI" below). `addClassTeacher`/`removeClassTeacher` (either role) stay strictly ADMIN-only (`requireClassAdmin`) — including granting the FACHLEHRER tier itself, confirmed as a deliberate admin-only decision — while `addSubjectTeacher`/`removeSubjectTeacher` stay deliberately self-service (`requireTeachesSubject` — any current teacher of the subject, no admin override) — that asymmetry is the whole point of having independent axes rather than one. The "keep ≥1" guardrail on `ClassTeacher` removal is role-aware, not a flat row count: removing an `ADMIN`-tier row is blocked only if it's the *last remaining admin* (`ClassUiResource#removeClassTeacher`, backed by `ClassTeacher#canRemove` — see `fragments/classTeachers.html`); a `FACHLEHRER`-tier row is never blocked by it; a class can have zero of those at any time. There is still no "delete a class only when you're its sole remaining teacher" cross-axis guardrail — each axis's guardrail stays local to itself.

`web/ClassOverviewResource` (`GET /classes/{id}/overview`, `requireClassAdmin`-gated) is the "See all final grades of Fächer and final grade of students" ADMIN capability — a read-only, class-wide table (students x every Subject, final grade + raw average), deliberately shallow: no drill-down into a Subject's categories/Leistungen/individual Grades, so an admin who doesn't personally teach a Subject still can't reach `/subjects/{id}` (unchanged, still `requireTeachesSubject`-gated) — only this aggregate view. It reuses `rest/ClassAveragesResource#averages` directly via CDI injection (a plain in-JVM method call, not HTTP) rather than recomputing the per-student-per-subject average — that method's own `isAdmin`-vs-`teachesSubject` row filtering already returns every Subject once the caller is confirmed an admin. `ClassAveragesResource` itself (`/api/school-classes/{classId}/averages`) predates this feature and was, until now, unused by any page — worth knowing if its shape ever needs to change, since `ClassOverviewResource` is now a real caller, not just the REST API's only consumer being an external client.

### Class & subject access UI: the `Teacher` directory, `addClassTeacher`/`addSubjectTeacher`
There's no persisted directory of teachers by email/name to pick a colleague from when sharing class access — `teacherSubject` is just an opaque OIDC `sub`. `domain/Teacher` closes that gap: a row per subject that's ever made an authenticated request, with a best-effort `email`/`displayName` (`Teacher#displayLabel()` prefers email, then displayName, then falls back to the raw subject) kept fresh on every sighting. It's shared/unowned data, the same category as `GradeScale` — not `OwnershipGuard`-scoped, any authenticated teacher can read the full list (a deliberate, mild trade-off: any teacher adding a co-owner to *their own* class can see who else has used this instance, not just people they already share a class with — acceptable for this app's typically-one-school self-hosted deployment model).

`security/TeacherDirectoryRecorder` populates it via an `@Observes AuthenticationSuccessEvent` listener (fired by Quarkus's own `HttpAuthenticator` for local FORM auth and OIDC alike — no per-mechanism plumbing needed) rather than a hand-rolled filter. Getting this right took three rounds of empirical fixes, all now documented in the class's own Javadoc, because none of the obvious approaches work: `@Blocking` cannot be placed on an `@Observes` method at all (rejected at build time as "not an entrypoint"); the event fires on the Vert.x event-loop thread, so `@Transactional` on the observer throws `BlockingOperationNotAllowedException`; and `CurrentUser`'s accessors are unsafe to call there too (their blocking `SecurityIdentity` association lookup throws `IllegalStateException("The current thread cannot be blocked")`) — dispatching that read alone onto a worker thread doesn't fix it either, since the per-request security context isn't propagated the way CDI request scope normally is, so it silently resolves as unauthenticated instead. The working shape: read `subject`/`email`/`displayName` synchronously from `AuthenticationSuccessEvent#getSecurityIdentity()` itself (a concrete resolved instance, not the lazy proxy — the `userinfo` attribute key is the same one `OidcUtils.setSecurityIdentityUserInfo` populates), then hand the actual DB write off to a `ManagedExecutor` worker thread using `QuarkusTransaction.requiringNew()` (not `@Transactional` — a fire-and-forget lambda doesn't go through the bean's own CDI proxy for the interceptor to apply). It deliberately does **not** fire under `%dev`/`%test`'s permit-all HTTP policy (no authentication is ever attempted there) — fine rather than a gap for `%test`, since tests seed exactly the `Teacher` rows they need directly via Panache; `%dev` gets its own fix instead (below), since a developer poking around locally has no such fixture to write.

`security/DevTeacherSeeder` (`@IfBuildProfile("dev")`, `@Observes StartupEvent`) closes the resulting `%dev` gap: without it, browsing around locally would never give the fixed dev-mode subject a `Teacher` row at all (nothing else ever fires the recorder there), and there'd be no way to conjure up a second local identity to test the add-a-co-owner flow against either. It seeds two rows on startup if missing — the effective dev-mode subject itself, and `<subject>-2` as a synthetic "colleague" (with a fake email/display name so `displayLabel()` has something to show) — then leaves them alone; not active under `%test`, for the same "don't make tests depend on ambient startup data" reason as the recorder note above. The dev-mode subject itself (`CurrentUser.DEV_USER_SUBJECT`, `"dev-user"`) is overridable per-developer via `DEV_USER_SUBJECT` (env var → `notenfuchs.dev-user-subject`, `%dev`-scoped only in `application.properties` — deliberately **not** read under `%test`, so a value sitting in a developer's shell can never change what deterministic test fixtures assume the fixed `%test` subject is) instead of editing/recompiling the Java constant.

`web/ClassUiResource#addClassTeacher`/`#removeClassTeacher` (`POST`/`DELETE /classes/{id}/teachers[/{teacherId}]`, ADMIN-only, either role) are named after `ClassTeacher` specifically, not generically "teacher" — this codebase has three teacher concepts (`Teacher` the directory, `ClassTeacher`, `SubjectTeacher`), and `web/SubjectUiResource#addSubjectTeacher`/`#removeSubjectTeacher` (`POST`/`DELETE /subjects/{id}/teachers[/{teacherId}]`) is the near-identical subject-level sibling this naming was chosen to stay unambiguous against. `addClassTeacher` takes a `role` form param (`ADMIN`/`FACHLEHRER`, defaulting to `FACHLEHRER` when blank — the lower-privilege default). Adding validates the picked subject is actually a known `Teacher` and not already a member (400 otherwise) in both; removing enforces "keep ≥1 admin" (`ClassTeacher`, role-aware — see above) / "keep ≥1 teacher" (`SubjectTeacher`) respectively (400, primarily defended by the template not rendering "Entfernen" next to a row that would violate it).

Self-removal is where the two diverge, precisely because of the self-service difference above. `removeClassTeacher` can hardcode its redirect: losing your last `ClassTeacher` row always means losing the whole class, so `HX-Redirect: /classes` is always correct. `removeSubjectTeacher` can't hardcode anything: losing your one `SubjectTeacher` row on *this* subject doesn't necessarily cost class access at all (you might still own the class, or teach another subject in it) - so it computes the redirect from `OwnershipGuard#hasClassAccess`, checked *after* the delete (same transaction - Hibernate's AUTO flush makes that query see the pending deletion, verified empirically against a real instance, not just assumed), picking `/classes/{schoolClass.id}` when access remains and `/classes` only when it doesn't.

`security/DevIdentitySwitchResource` (`@IfBuildProfile("dev")`, `/dev/identity`) is what actually makes `DevTeacherSeeder`'s synthetic second colleague useful for testing either of the above: without it, a developer could see `<subject>-2` in an add-teacher dropdown but never *act* as them, since `%dev`/`%test` otherwise always resolve to the one fixed identity. It writes `CurrentUser.DEV_IDENTITY_COOKIE`, which `CurrentUser#effectiveSubject()` only ever trusts when `CurrentUser.DEV_IDENTITY_SWITCH_ACTIVE_PROPERTY` is active (`%dev`-scoped only, same as `DEV_USER_SUBJECT_PROPERTY` above) - a real security boundary, not just a UX default, since honoring a client-supplied cookie as an identity override anywhere else would be a straightforward auth bypass. Linked from `base.html`'s existing "Dev-Modus" nav badge.

## Conventions & gotchas

- **Flyway owns the schema**, not Hibernate. `quarkus.hibernate-orm.database.generation=validate`. Schema changes go in a new `src/main/resources/db/migration/V*__*.sql` migration — never rely on auto-DDL.
- Grade-calc changes must come with tests in `GradeServiceTest` (covers factor weighting, category combination, empty-category normalization, both rounding modes at the .5 boundary).
- Points→grade conversion changes must come with tests in `PointsConversionServiceTest` (band boundaries at exact thresholds, order-independence, scale-agnosticism, the no-band-matches fallback, the default-bands seeding, and both `RoundingMode`s — `COMMERCIAL` vs. `IN_FAVOR_OF_STUDENT` — on both scale directions). Any Qute template that reads a lazy entity association (e.g. `subject.gradeScale`) from a `TemplateInstance` returned by an `@Transactional` web endpoint must force-initialize it first (`Hibernate.initialize(...)`) — the transaction's Hibernate session is already closed by the time Qute actually renders the response, so an untouched lazy proxy throws `LazyInitializationException` (see `SubjectUiResource#categoryFragment`).
- Free-text student names by design — the teacher decides what to enter (real name or Kürzel). No PII assumptions baked into the model.
- The grade-entry grid (`GradeGridResource` + `notenfuchs.js`) is a spreadsheet-style UI (students as rows, assessments as columns, tab/enter between cells, per-cell autosave) — the thing existing tools do badly, and the main reason to browser-test a change instead of trusting `GradeServiceTest` alone.
- UI/grid changes should come with a Playwright IT in `src/test/java/de/notenfuchs/e2e` (`GradeGridE2EIT`) — run via `./mvnw verify`, not `./mvnw test` (see Commands). These drive the real server-rendered pages through a browser (`quarkus-playwright`, Dev Services container - no local browser install needed) against a Testcontainers Postgres, so they need Docker. Each test creates its own uniquely-named class/subject/student rather than relying on a clean DB, since there's no reset between tests.
- The README's screenshots (`screenshots/*.png`) are generated, not hand-taken, by `ReadmeScreenshotIT` (`src/test/java/de/notenfuchs/e2e`) — same Playwright/Dev Services machinery as the other browser ITs, but logging in via `LocalAuthTestProfile` (real "Angemeldet als lehrer" nav, not the `%test` permit-all bypass's "Dev-Modus" badge) and using made-up placeholder names, since these screenshots are meant to look like a real deployment. It's excluded from the default `./mvnw verify` run (`pom.xml`'s failsafe `<excludes>`) — it's a manual, on-demand content-generation step, not a regression test — run explicitly via `./mvnw verify -Dit.test=ReadmeScreenshotIT` (which overrides the exclude), then commit `screenshots/` only if `git diff --stat screenshots/` shows an actual change. Each captured PNG is decoded/re-encoded once through `ImageIO` before being written, so re-running the generator with no visible UI change reproduces byte-identical files (no incidental metadata, no encoder-noise diffs) instead of needing an external image-optimizer tool installed. `V3__seed_demo_class.sql` seeds the class's `halfYearCutoff` and its assessments' `date`s relative to `CURRENT_DATE` (so a fresh install always looks "recent"), which would otherwise make the grid screenshot drift with the calendar - not just the displayed "Halbjahres-Stichtag" text but potentially the Ohne-Datum/1.-HJ/2.-HJ column split itself; `ReadmeScreenshotIT` overwrites the cutoff and every assessment date to fixed calendar dates via the app's own edit UI right after reaching the demo class, so the result stays deterministic regardless of which day it's run on. If the UI changes meaningfully (new page section, restyled grid, etc.), re-run this IT and review the diff rather than hand-editing the PNGs.
- Roster CSV import/export (`CsvRosterService`) has to cope with real German Excel output: semicolon (not comma) delimiter and Windows-1252 (not UTF-8) encoding. Export always writes the Excel-friendly dialect (UTF-8 with a BOM, semicolon-delimited); import sniffs the delimiter from the header line and falls back UTF-8 → Windows-1252 on decode failure. Import is two-step and stateless — upload renders a preview (new vs. duplicate per row, by exact name match) whose confirm form round-trips the parsed names as hidden inputs rather than a server-side session; only the confirm POST persists anything. Changes here need unit tests in `CsvRosterServiceTest` (delimiter/encoding/line-ending/header/quoting/round-trip) plus a Playwright IT (`RosterImportExportE2EIT`) if the web flow changes.
- Access-control changes (new endpoint scoped by class/subject, new entity type) need coverage in `OwnershipGuardIT` (`src/test/java/de/notenfuchs/security`, a `@QuarkusTest` + `@TestTransaction` IT — needs Docker/Postgres, run via `./mvnw verify`) asserting cross-tenant isolation (a foreign id 404s, an owned id resolves) *and* the class-access-vs-teaches-subject distinction (a collaborator who teaches one subject in a class can reach `requireClassAccess*` but still 404s from `requireTeaches*` on a different subject in the same class), plus `CurrentUserTest` if `CurrentUser`'s fallback logic changes.
- Class duplication changes need coverage in `ClassDuplicationIT` (`src/test/java/de/notenfuchs/web`, a `@QuarkusTest` IT driving the real endpoint over plain `java.net.http.HttpClient` — no RestAssured dependency in this project) asserting exactly what gets copied (Subjects/GradeCategories/Students) versus what doesn't (Assessments/Grades), that both classes stay independently editable, and that a foreign source class 404s; plus a Playwright IT (`ClassDuplicationE2EIT`) if the web flow changes.
- Class-teacher-management changes (`addClassTeacher`/`removeClassTeacher`, the `Teacher` directory) need coverage in `ClassTeacherIT` (`src/test/java/de/notenfuchs/web`, add/remove flows for both `ClassTeacherRole`s, the role-aware last-admin guardrail — blocked when removing the sole admin even with a `FACHLEHRER` row present, never blocked for a `FACHLEHRER` row itself — non-admin 404s, self-removal's `HX-Redirect` header — "colleague" teachers are seeded directly as `Teacher` rows via Panache, not through a real login) plus a Playwright IT (`ClassTeacherE2EIT`, which must explicitly select the `ADMIN` role in the add form when a test wants a second co-admin, since the default is `FACHLEHRER`) if the web flow changes. `TeacherDirectoryRecorder` itself needs its own `TeacherDirectoryRecorderIT` (`src/test/java/de/notenfuchs/security`) using `LocalAuthTestProfile` and a real `/j_security_check` login exactly like `LoginIT` — the normal `%test` profile's permit-all policy means `AuthenticationSuccessEvent` never fires at all, so this can't be exercised any other way; poll briefly for the row rather than asserting immediately, since the write is deliberately fire-and-forget off the request thread.
- Subject-teacher-management changes (`addSubjectTeacher`/`removeSubjectTeacher`) need coverage in `SubjectTeacherIT` (`src/test/java/de/notenfuchs/web`, mirrors `ClassTeacherIT`'s style) — beyond the add/remove/guardrail cases `ClassTeacherIT` already covers, this axis specifically needs a case proving a *non-admin* collaborator who merely teaches the subject can add/remove another teacher (self-service, not admin-gated — the whole point of this axis being separate from `ClassTeacher`, and specifically **not** overridable by an admin who doesn't teach the subject — see `OwnershipGuardIT#requireTeachesSubject_adminNotTeachingSubject_stillThrowsNotFound_noAdminOverride`), and both branches of the self-removal redirect (`/classes/{id}` when the actor still has class access some other way, `/classes` when they don't) — plus a Playwright IT (`SubjectTeacherE2EIT`) if the web flow changes.
- Changes to the three-tier split itself (`ClassTeacherRole`, `requireClassTeacher`/`requireCanDeleteSubject`, `addSubject`/`deleteSubject`'s gates) need coverage in `OwnershipGuardIT` (role-aware predicate tests) and `SubjectStructuralAccessIT` (`src/test/java/de/notenfuchs/web` — HTTP-level: a subject-only Fachlehrer rejected from both `addSubject` and `deleteSubject` even for their own Subject, a class-level Fachlehrer allowed to add/delete only Subjects they teach, an admin allowed to delete any Subject regardless of teaching status). `ClassOverviewResource` changes need coverage in `ClassOverviewIT` (admin sees every Subject's final grade for every student; a class-level Fachlehrer and a subject-only Fachlehrer both 404; foreign class 404s). The class-list Ändern/Löschen gating (`adminClassIds`) needs coverage in `ClassTeacherIT#classList_hidesRenameAndDeleteForClassesWhereViewerIsNotAdmin`. Per-Fach teacher display changes (`ClassUiResource#attachTeacherLabels`, `Subject#primaryTeacherLabel`/`#otherTeacherLabels`) need coverage in `SubjectTeacherDisplayIT` (`src/test/java/de/notenfuchs/web` — sole teacher shows no collapsible, 2+ teachers show the first plus a `+N weitere` `<details>` with the rest, and the display appears even for a Subject the viewer doesn't teach). The "Fachlehrer mit Klassenzugriff" section (`ClassUiResource#transitiveTeachers`) needs coverage in `TransitiveTeacherIT` (`src/test/java/de/notenfuchs/web` — a subject-only Fachlehrer with no `ClassTeacher` row is listed with their Subject name, a teacher holding both a `ClassTeacher` row and a `SubjectTeacher` row is excluded, and the section is absent entirely for a non-admin viewer).
- Halbjahr split-view changes (`GradeGridResource`, `HalfYearAssessmentPartitioner`, `GridPage/gridHalfYear.html`) need unit tests in `HalfYearAssessmentPartitionerTest` (boundary-date classification, undated), the HTTP-level `GradeGridHalfYearIT` (`src/test/java/de/notenfuchs/web`, per-half averages against a date-filtered subset, xlsx structure, null-cutoff backward compatibility) and a Playwright IT (`GradeGridHalfYearE2EIT`) if the grid/web flow changes. When driving the "add Leistung" htmx form twice in a row in a Playwright test, wait for the first row to actually appear (e.g. assert the row count) before filling the second submission — otherwise the second `fill()` races the first request's still-in-flight `outerHTML` swap and silently lands on the about-to-be-replaced form (see `GradeGridHalfYearE2EIT`).

## Deployment note
Self-hosted on a VPS. Sensitive data (student names + grades) lives in Postgres. Recommended at-rest protection is a LUKS-encrypted volume for the DB (protects against snapshot/disk theft; not against a live host compromise). Pseudonymization via `displayName` is the cheapest strong safeguard.
